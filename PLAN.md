# План воспроизведения MVP: ИИ-ассистент для web-дисциплин (RAG + faithfulness)

**Источник:** доклад `speech_v5.pdf` + `project_plan.md`, ИСТ ДГТУ, 09.06.2026
**Цель воспроизведения:** проверить главную гипотезу проекта — что RAG-ассистент, привязанный к корпусу учебных материалов, отвечает с **faithfulness ≥ 0.9** и корректно отказывается, когда ответа в корпусе нет.
**Среда:** Google Colab (T4 GPU)
**Глубина:** MVP / proof-of-concept

---

## 1. Гипотезы, которые проверяем

| № | Гипотеза | Метрика | Целевое значение |
|---|----------|---------|------------------|
| H1 | RAG отвечает достоверно (по фрагментам корпуса) | `faithfulness` (RAGAS) | ≥ 0.90 |
| H2 | Извлекаются релевантные фрагменты | `context_precision`, `context_recall` | ≥ 0.80 |
| H3 | Ответ соответствует вопросу | `answer_relevancy` | ≥ 0.85 |
| H4 | Ассистент отказывается отвечать, если корпус не содержит ответа | accuracy on OOD-set | ≥ 0.90 |
| H5 | «Красные линии» (оценка/решение за студента) блокируются | rule-based, 100% | 100% |

OOD-set = набор вопросов, ответ на которые **отсутствует** в корпусе (например, «как настроить FastAPI?» когда корпус только HTML/CSS).

## 2. Архитектура MVP (упрощённая трёхзвенная из доклада)

```
                ┌──────────────────────────────────────────────┐
                │              Colab Notebook                  │
                │                                              │
   запрос  ──►  │  guard_rules ──► (block если red zone)       │
                │       │                                      │
                │       ▼                                      │
                │  e5-multilingual ──► query embedding         │
                │       │                                      │
                │       ▼                                      │
                │  ChromaDB (top-k=4) ──► фрагменты + sources  │
                │       │                                      │
                │       ▼                                      │
                │  similarity gate ── (если max_sim < τ → "не знаю")
                │       │                                      │
                │       ▼                                      │
                │  Qwen2.5-7B-Instruct + RAG-prompt            │
                │       │                                      │
                │       ▼                                      │
                │  ответ + цитаты [doc_id, заголовок, url]     │
                └──────────────────────────────────────────────┘
                                  │
                                  ▼
                       RAGAS evaluation
                       (faithfulness, recall, precision)
```

**Ключевые архитектурные ограничения (как в докладе):**
- В коде физически нет функции «выставить балл» / «дать педагогический комментарий».
- Каждый ответ обязан содержать ссылки `[1], [2]`, иначе генерация отклоняется.
- При max similarity < `τ = 0.55` возвращается «Не нашёл этого в материалах курса».

## 3. Технологический стек MVP

| Слой | Выбор | Обоснование |
|------|-------|-------------|
| Среда | Google Colab + T4 GPU | бесплатно, GPU 15 GB |
| LLM | `Qwen/Qwen2.5-7B-Instruct` (4-bit через `bitsandbytes`) | сильный мультиязычный, влезает в T4 |
| Эмбеддинги | `intfloat/multilingual-e5-large` | как в исходном плане проекта |
| Vector store | `chromadb` (in-memory) | минимум кода, как в плане |
| Splitter | LangChain `RecursiveCharacterTextSplitter`, chunk=500, overlap=80 | по плану |
| Source corpus | MDN: фрагменты HTML/CSS/JS (~30–50 страниц) | публичные, CC-BY-SA |
| Eval | `ragas` + `datasets` | стандарт de-facto для RAG в 2025–2026 |
| Judge-модель для RAGAS | тот же Qwen2.5-7B локально (LangChain wrapper) | без внешних API |

## 4. Этапы реализации (для MVP в один день)

| Этап | Артефакт | Время |
|------|----------|-------|
| 1. Окружение в Colab | блок установки, проверка GPU | 5 мин |
| 2. Сбор MDN-корпуса | `data/mdn_corpus.jsonl` | 10 мин |
| 3. Индексация (e5 + Chroma) | persistent collection | 10 мин |
| 4. RAG-цепочка + guard rules | функция `ask(question)` | 20 мин |
| 5. Демо-сценарии из доклада | 4 кейса: flexbox, CSS-линт, Composition API (эскалация), оценка (блок) | 10 мин |
| 6. Eval-набор (15–25 Q/A) | `eval/qa.jsonl` (15 in-corpus + 10 OOD) | 15 мин |
| 7. Прогон RAGAS | таблица метрик + вывод по H1–H4 | 15 мин |
| 8. Анализ ошибок | топ-3 проваленных кейса, что улучшать | 10 мин |

Итого ~1.5–2 часа в Colab.

## 5. План эскалации (как в докладе, Q&A №8)

Если faithfulness < 0.9 на первом прогоне:
1. Увеличить `top_k` с 4 до 6.
2. Добавить **cross-encoder reranker** (`BAAI/bge-reranker-v2-m3`).
3. Понизить `temperature` LLM до 0.1.
4. Усилить системный промпт: «отвечай **только** утверждениями из контекста».
5. Если после трёх итераций цель не достигнута — публикуем как ограничение MVP и сужаем темы.

---

## 6. Что НЕ входит в этот MVP (по сравнению с полным проектом)

- Frontend (Vue) — уже есть в `AI-assistent-IST-DGTU-demo.zip`, к нему позже подключим backend через ngrok.
- FastAPI-сервис и Docker — следующий шаг после валидации RAG-ядра.
- Авторизация, bcrypt, журнал PII — не нужно для оффлайн eval.
- Дашборд аналитики затруднений — отдельный модуль (агрегация запросов).
